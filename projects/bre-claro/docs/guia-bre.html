1Contexto

O BRE (Business Rules Engine) do Salesforce Industries permite expressar regras de decisão/cálculo de forma declarativa, sem Apex "dono" das regras. Os dois blocos centrais são:

BlocoMetadataRuntime (objeto)Papel
Decision Matrix DecisionMatrixDefinition CalculationMatrix / CalculationMatrixVersion / CalculationMatrixRow Tabela de lookup: chave(s) de entrada → valor(es) de saída
Expression Set ExpressionSetDefinition ExpressionSet / ExpressionSetVersion Orquestrador: encadeia lookups e cálculos, mantém contexto compartilhado
💡
Mental model: a Definition é a metadata (versionável por deploy). As linhas da matriz são dados (CalculationMatrixRow), carregados em runtime via API/Bulk — não vão no deploy.

O Expression Set "navega" entre matrizes sem relacionamento físico entre elas: ele mantém um contexto de variáveis e a saída de um step alimenta o próximo (ex.: preço base → política → fidelidade → bundle → preço final).

2Pré-requisitos

  • BRE habilitado na org (feature Salesforce Industries / Revenue Cloud).
  • Salesforce CLI (sf) autenticado na org alvo:
sf org login web -a org-demo
sf org display -o org-demo
  • Projeto SFDX com a pasta force-app/main/default/{decisionMatrixDefinition,expressionSetDefinition}.

3Parte A — Criar um BRE do zero

✅
Ordem à prova de trava para POC do zero: faça o deploy só das matrizes primeiro, carregue as linhas (desabilita → load → reabilita uma versão por vez) e só então deploye o Expression Set. Isso evita a armadilha #22 (não dá pra desabilitar matriz referenciada por ES ativo) já na carga inicial.
  1. 1
    Definir a Decision Matrix
    Crie force-app/main/default/decisionMatrixDefinition/<Nome>.decisionMatrixDefinition-meta.xml. Cada coluna é Input (chave) ou Output (resultado). 
    <?xml version="1.0" encoding="UTF-8"?>
<DecisionMatrixDefinition xmlns="http://soap.sforce.com/2006/04/metadata">
    <label>Real Preco Matrix</label>
    <type>Standard</type>
    <versions>
        <fullName>RealPrecoMatrix_V1</fullName>
        <columns>
            <columnType>Input</columnType>
            <dataType>Text</dataType>
            <displaySequence>1</displaySequence>
            <isWildcardColumn>false</isWildcardColumn>
            <name>PRODUTO</name>
        </columns>
        <!-- demais colunas Input (MODALIDADE, DURACAO, CANAL)... -->
        <columns>
            <columnType>Output</columnType>
            <dataType>Number</dataType>
            <displaySequence>5</displaySequence>
            <isWildcardColumn>false</isWildcardColumn>
            <name>PRECO_BASE</name>
        </columns>
        <status>Active</status>
        <versionNumber>1</versionNumber>
    </versions>
</DecisionMatrixDefinition>
    💡
    Coluna wildcard: use isWildcardColumn=true para linhas "default/catch-all". Exige <wildcardValue>*</wildcardValue> declarado após <name>. Limite: apenas UMA coluna wildcard por matriz. Hierarquia mais profunda (cidade > UF > nacional) se faz com lookups em camadas + COALESCE no ES.
    ⚠️
    Tipo numérico: coluna de matriz usa dataType=Number (sem rótulo de moeda). Variável do ES usa Currency (o enum ExpsSetDataType não aceita Number). Padrão validado: coluna da matriz Number + variável do ES Currency.
  2. 2
    Definir o Expression Set (orquestrador)
    Crie .../expressionSetDefinition/<Nome>.expressionSetDefinition-meta.xml. Três partes importantes: steps (sequência), variables (contexto), e referências às matrizes. 
    <!-- Step 1: lookup numa matriz -->
<steps>
    <actionType>GetOutputsFromDecisionMatrix</actionType>
    <decisionTable>
        <decisionTableName>RealPrecoMatrix</decisionTableName>
        <type>DecisionMatrix</type>
    </decisionTable>
    <name>Step1_Base</name>
    <sequenceNumber>1</sequenceNumber>
    <stepType>BusinessKnowledgeModel</stepType>
</steps>

<!-- Step 2: cálculo / composição -->
<steps>
    <actionType>AssignParameterValues</actionType>
    <assignment>
        <assignedParameter>PRECO_POS_POLITICA</assignedParameter>
        <expression>RealPrecoMatrix__PRECO_BASE * RealPoliticaMatrix__FATOR</expression>
    </assignment>
    <name>Step_Calculo</name>
    <sequenceNumber>3</sequenceNumber>
    <resultIncluded>true</resultIncluded>
    <stepType>BusinessKnowledgeModel</stepType>
</steps>
    ⚠️
    Limite rígido: máximo de 200 steps por versão de ES. Cada lookup e cada cálculo conta como 1 step. Input compartilhado: declare a variável de entrada uma única vez mesmo que alimente várias matrizes — o casamento é feito por nome de coluna em runtime.
  3. 3
    Deploy da metadata
    sf project deploy start -o org-demo \
  -d force-app/main/default/decisionMatrixDefinition/RealPrecoMatrix.decisionMatrixDefinition-meta.xml \
  -d force-app/main/default/expressionSetDefinition/RealPreco.expressionSetDefinition-meta.xml
  4. 4
    Carregar as linhas da matriz (dados)
    Monte um CSV com apenas estas colunas (não inclua Name — ver armadilha #1):
    CalculationMatrixVersionId,InputData,OutputData
ID_MATRIZ_EXEMPLO,"{""PRODUTO"":""COM_INT_RES_1GB"",""MODALIDADE"":""INDIVIDUAL"",""DURACAO"":""0"",""CANAL"":""DIGITAL""}","{""PRECO_BASE"":199.9}"

    Primeiro obtenha o Id da versão:

    sf data query -o org-demo \
  -q "SELECT Id, CalculationMatrix.Name FROM CalculationMatrixVersion WHERE CalculationMatrix.Name LIKE 'Real%'"

    Carga via Bulk API (CSV em LF — ver armadilha #8):

    sf data import bulk -o org-demo -s CalculationMatrixRow -f real_preco.csv --wait 30
    ⚠️
    Carregar linhas exige a versão da matriz DESABILITADA. Numa matriz recém criada e ativa, desabilite a versão antes do load e reabilite depois. Habilite uma versão por vez — até matrizes diferentes não podem ser habilitadas na mesma transação (armadilha #4).
  5. 5
    Ativar e validar
    # Confirma versão ativa
sf data query -o org-demo \
  -q "SELECT COUNT() FROM ExpressionSetVersion WHERE ExpressionSet.ApiName='RealPreco' AND IsActive=true"

# Invoca via REST
curl -s "$INSTANCE_URL/services/data/v60.0/actions/custom/runExpressionSet/RealPreco" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -X POST -d '{"inputs":[{"PRODUTO":"COM_INT_RES_1GB","MODALIDADE":"INDIVIDUAL","DURACAO":"0","CANAL":"DIGITAL"}]}'

    Resposta esperada: isSuccess: true e outputValues com os valores calculados. Teste sempre o caminho no-match também (armadilha #10).

4Parte B — Alterar um BRE existente

Caso 1 — Alterar linhas de uma matriz (com janela curta)

💡
Linha é DADO, não estrutura. Para incluir/editar/excluir linhas você NÃO precisa recriar a matriz nem criar nova versão. A única regra é que a plataforma não permite DML com a versão habilitada.
1. Desativar a versão do ES que referencia a matriz  (IsActive=false)
2. Deletar a versão do ES
3. Desabilitar a versão da matriz                     (IsEnabled=false)
4. Inserir/atualizar/deletar as linhas (Bulk API)
5. Reabilitar a versão da matriz                      (IsEnabled=true) — UMA POR VEZ
6. Redeployar o Expression Set

Apex anônimo para os passos 1–3 e 5:

// 1+2. desativa e deleta versão do ES
List<ExpressionSetVersion> evs = [SELECT Id, IsActive FROM ExpressionSetVersion
                                  WHERE ExpressionSet.ApiName = 'RealPreco'];
for (ExpressionSetVersion v : evs) v.IsActive = false;  update evs;
delete evs;

// 3. desabilita versão da matriz
CalculationMatrixVersion mv = [SELECT Id, IsEnabled FROM CalculationMatrixVersion WHERE Id = '0lNak...'];
mv.IsEnabled = false;  update mv;
# 4. carrega as novas linhas
sf data import bulk -o org-demo -s CalculationMatrixRow -f novas_linhas.csv --wait 30

# 6. redeploy reativa o ES
sf project deploy start -o org-demo -d force-app/main/default/expressionSetDefinition/RealPreco.expressionSetDefinition-meta.xml

Caso 1-B — Trocar linhas SEM downtime (blue-green de versão) — recomendado

1. Cria a versão V2 da matriz (desabilitada)        — a V1 segue servindo o ES
2. Popula a V2 com o conjunto COMPLETO de linhas     — DML livre (V2 desabilitada)
3. Ativa/habilita a V2                                — swap automático: V1 aposentada, V2 sobe
                                                        (o ES resolve pela versão habilitada, sem redeploy)
✅
Rollback trivial: reative a V1. O ES não muda — ele referencia a matriz por nome e usa a versão habilitada em runtime.
Pela UI do Decision Matrix Designer, "Criar nova versão" clona as linhas da versão atual, permitindo editar só o delta manualmente.

Caso 2 — Alterar a estrutura (colunas da matriz ou steps do ES)

Mudança estrutural = nova versão. Edite o -meta.xml incrementando versionNumber / fullName (..._V2), ajuste rank/status e faça deploy. A versão anterior é preservada para rollback.

Caso 3 — Versionar/publicar nova regra do ES

  • Suba a nova ExpressionSetVersion com status=Active e versionNumber maior.
  • Para remover: IsActive=false → delete (nessa ordem — não dá pra deletar versão ativa).

6Checklist de entrega

  • DecisionMatrixDefinition por domínio (uma responsabilidade por matriz)
  • ExpressionSetDefinition com steps ≤200 e variáveis de contexto nomeadas
  • Deploy da metadata sem erros
  • Linhas carregadas via Bulk (sem coluna Name, CSV em LF)
  • Versão do ES ativa (IsActive=true por ApiName)
  • Teste funcional via runExpressionSet — caminho match E no-match passando
  • Linha/coluna wildcard de fallback prevista para todo lookup
  • Invocação sempre em bulk (lista de inputs numa chamada) na integração

📐
Princípio de arquitetura: Apex fica na borda (integração, bulkificação, montagem de contexto). As regras ficam declarativas no BRE. Isso reduz drasticamente o "Apex dono de regra" e melhora governança e versionamento.

Padrão recomendado: indexar valores absolutos por uma chave rica (PRODUTO × MODALIDADE × DURACAO × OPERACAO × ...) + ajustes aditivos em R$, em vez de multiplicar fatores percentuais (no caso da Claro, IsPercentDiscount__c=false em 100%).
